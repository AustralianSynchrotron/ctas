\subsubsection{Detailed Description}
Template-\/oriented library for CLI parsing. \begin{DoxyAuthor}{Author}
antonmx $<$\href{mailto:antonmx@asw062.synchrotron.org.au}{\tt antonmx@asw062.synchrotron.org.au}$>$ 
\end{DoxyAuthor}
\begin{DoxyDate}{Date}
Mon Mar 16 13:39:22 2009
\end{DoxyDate}
This is just another command line interface parsing library. I wrote it because could not find any other one (among dozens or even hundreds) which could satisfy me. The architecture of this library was strongly influenced by the libpopt (part of the rpm package) which I used for several years. The characteristic features of my library are as follows:
\begin{DoxyItemize}
\item Object-\/oriented approach and architecture: C++/STL.
\item Cross-\/platform approach: compiles even on MS Windows.
\item No external libraries needed.
\item Can parse both arguments and options.
\item Template-\/based architecture: the parsing method is defined by the type of the variable to be updated.
\item Easy inclusion of any new types.
\item Possibility to redefine the standard parse procedures for any type.
\item Possibility to automatically generate help, usage and verbose options.
\item Possibility to automatically generate manual pages.
\item The entries to the option table appear in the help and usage messages and manual pages exactly as they were added to the table: no argument/option and/or alphabetical sort is used.
\end{DoxyItemize}

Writing this library I do assume a few things about the CLI logic:


\begin{DoxyItemize}
\item The element to be parsed may have only one meaning (certain type and the variable to be updated). This assumption is different from the libpopt where any option could have it's argument representing different types. I think it may provoke errors in the CLI. However you can overcame this limitation defining your own type or redefining the parsing method of a predefined one.
\item The option's argument may consist only of one CLI word in the phrase. It is not difficult to write a support for the multi-\/element option's argument, but it will make the command line harder to read and understand.
\item The option may NOT be defined as \char`\"{}having or not having the argument\char`\"{}. Also different from the libpopt behaviour for the same reasons.
\item The same option name (short or long) cannot be used in different entries.
\item The same element can be used more than once in the CLI input phrase only if the corresponding variable is an std::vector. In this case each next appearance of the element in the CLI phrase will add the element to the end of the vector.
\item The CLI input processing is case sensitive. 
\end{DoxyItemize}
